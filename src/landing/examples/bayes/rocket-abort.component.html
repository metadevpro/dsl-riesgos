Example 2. Rocket Abort Launch at T-10

In the final second before launch, the rocket's software automatically checks several critical systems. If any of the sensors detects an anomaly, the system can trigger an automatic abort at T-10, halting the launch before ignition is complete.

The problem is that an alarm does not always indicate an actual physical fault. Sensors can generate false positives due to electrical noise, anomalous readings, temporary deviations or software errors. Therefore, when an alert is triggered, the flight controller needs to answer a key question:

If the abort system has been activated, what is the probability that there is actually a physical fault?

This example shows how a Bayesian network allows a binary alert — “abort” or “no abort” — to be transformed into a probabilistic estimate useful for decision-making

Context of the case

The model depicts a scenario inspired by real-world launches, where an anomalous reading at T-10 can temporarily halt the mission. In such scenarios, the objective is not merely to detect the alarm, but to assess whether that alarm corresponds to:

  • A genuine physical fault in the vehicle.
  • A false positive generated by the sensor system.
  • A combination of several weak signals.
  • Strong evidence, enough to warrant a full inspection.

In a risk analysis application, this example demonstrates the value of the DSL developed in the final year project: the user can visually construct the network, define the conditional probability tables and observe how the probability of a real failure changes when new evidence is introduced.

The network contains five main nodes:

  • Real System Failure: There is a real physical failure in one of the critical systems at T-10.
  • Propellant Sensor: The propellant sensor system reports an anomaly.
  • EC Sensor: The engine control system sensor reports an anomaly.
  • T-0 Alert: The software activates automatic abort.
  • Failure is real: The alert actually corresponds to a physical failure and not a false positive.

Probabilities defined in the CPTs

Node 1 · RealSystemFailure

This node represents the prior probability that an actual physical fault exists at T-10.

State Probability Meaning
Yes 0.8% Initial probability of a real fault in one of the critical systems.
No 99.2% Complementary: no real physical fault exists.

Node 2 · PropellantSensor

This node models the response of the propellant sensor depending on whether a real fault exists.

RealSystemFailure P(PropellantSensor = yes) Interpretation
Yes 90% If there is a real fault, the sensor detects it in 90% of cases.
No 4% If there is no real fault, the sensor can produce a false positive in 4% of cases.

This table separates two important concepts:

  • Sensitivity: ability to detect a fault when one exists.
  • False positive: probability of triggering an alarm even when there is no real fault.

Node 3 · ECSensor

The engine control system sensor also depends on the existence of a real fault.

RealSystemFailure P(ECSensor = yes) Interpretation
Yes 95% If there is a real fault, the ECS sensor detects it with high probability.
No 3% If there is no real fault, a false alarm can occur in 3% of cases.

The ECS sensor is modelled as more sensitive and with a lower false positive rate than the propellant sensor.

Node 4 · T10Alert

This node represents the automatic decision made by the software. The alert is triggered with high probability if at least one of the sensors reports an anomaly.

PropellantSensor ECSensor P(T10Alert = yes) Interpretation
Yes Yes 99% Both sensors alarm; the abort is triggered almost with certainty.
Yes No 95% Only the propellant system alarms.
No Yes 93% Only the ECS system alarms.
No No 1% An abort may occur for other reasons: electrical noise, a bug or an external signal.

This CPT represents logic similar to an OR gate, but with uncertainty. No deterministic rule is enforced; instead, it allows the imperfections of the software and the environment to be modelled.

Node 5 · FailureIsReal

This node summarises whether the observed alert actually corresponds to a physical fault.

T10Alert P(FailureIsReal = yes) Interpretation
Yes 15% With an active alert, there is a 15% probability that the fault is real.
No 0.3% Without an alert, it is very unlikely that an undetected physical fault exists.

This node is especially useful because it summarises the practical value of the model: the network does not only detect an alarm, it estimates whether that alarm should be considered a real fault or a false positive.
